### WR-01: Raw database errors can reach Worker logs

**Classification:** WARNING
**File:** `apps/worker/src/db.rs:23`
**Issue:** The db-ping handler wraps PostgreSQL parse/connect/query failures with `format!("pg ...: {e:?}")` at lines 23, 29, and 43. With workers-rs `#[event(fetch)]`, returned errors are logged via `console_error!("{}", &e)`, and line 35 also logs the raw connection-task error. That bypasses the source grep in `.github/workflows/pow.yml` and can expose operational DB details such as user names, hostnames, SQLSTATEs, or authentication text in Worker logs.
**Fix:** Return sanitized responses and avoid logging raw `tokio_postgres::Error` values. For example:

```rust
fn db_unavailable() -> Result<Response> {
    Response::error("database unavailable", 503)
}

let config = match PgConfig::from_str(&conn_string) {
    Ok(config) => config,
    Err(_) => return db_unavailable(),
};

let (client, connection) = match config.connect_raw(socket, NoTls).await {
    Ok(parts) => parts,
    Err(_) => return db_unavailable(),
};

wasm_bindgen_futures::spawn_local(async move {
    if connection.await.is_err() {
        console_log!("pg connection task ended");
    }
});
```

### WR-02: Worker release profile is ignored by Cargo

**Classification:** WARNING
**File:** `apps/worker/Cargo.toml:26`
**Issue:** `[profile.release]` inside a non-root workspace package is ignored by Cargo. The review commands emit: `profiles for the non root package will be ignored`. This makes the worker-specific `opt-level = "s"`, `lto = true`, and `strip = "symbols"` block dead configuration, so future size or release assumptions can silently diverge from what Cargo actually uses.
**Fix:** Move effective profile settings to the workspace root. For example, keep shared release settings in root `Cargo.toml` and add any worker-only override there:

```toml
[profile.release]
opt-level = "s"
lto = "thin"
codegen-units = 1
strip = "symbols"

[profile.release.package.bloclawd-worker]
opt-level = "s"
```

Then remove the ignored `[profile.release]` block from `apps/worker/Cargo.toml`.

### WR-03: CI does not run tests or clippy for all new Rust crates

**Classification:** WARNING
**File:** `.github/workflows/pow.yml:40`
**Issue:** The workflow builds the workspace, runs `cargo test -p pow`, runs clippy only for `pow` and `xtask`, and runs event-schema tests for ts-rs drift. It never runs `cargo test -p bloclawd-worker`, so `apps/worker/tests/db_ping_contract.rs` is not enforced in CI. It also excludes `event-schema` and `bloclawd-worker` from the clippy gate even though they are part of this phase's new source surface.
**Fix:** Expand CI to cover the new crates:

```yaml
- name: cargo test --workspace
  run: cargo test --workspace --locked
- name: cargo clippy --workspace
  run: cargo clippy --workspace --locked -- -D warnings
```

If wasm-target clippy is required for the Worker, install `wasm32-unknown-unknown` and add a dedicated `cargo clippy -p bloclawd-worker --target wasm32-unknown-unknown --locked -- -D warnings` step.

### WR-04: Canonicalization docs claim a shared helper before pow uses it

**Classification:** WARNING
**File:** `spec/payload-canonical.md:12`
**Issue:** The spec says that after Phase 1.5 both Rust CLI and Rust Worker sides call `crates/event-schema::canonical_bytes`, but `crates/event-schema/src/jcs.rs:2` says `crates/pow` still keeps its inline `serde_jcs` call until Phase 2. The reviewed `apps/worker/src/lib.rs` also only has a compile-time witness to `crates/pow::verify`; runtime verification is not wired yet. This leaves the protocol doc overstating the current source-of-truth boundary and can hide future drift between `pow::payload_hash` and `event_schema::canonical_bytes`.
**Fix:** Either refactor `crates/pow` now to call `event_schema::canonical_bytes`, or update the spec language to reflect the actual interim state:

```markdown
After Phase 1.5, `crates/event-schema::canonical_bytes` is the Worker/CLI target helper.
`crates/pow` still uses the same pinned `serde_jcs = "0.2"` dependency inline until the
Phase 2 verifier refactor replaces that call.
```
