# Pitfalls Research

**Domain:** Anonymous community telemetry + Rust Cloudflare Worker + public R2 dashboard
**Researched:** 2026-04-30
**Updated:** 2026-04-30 - Phase 1.5 Rust Worker migration complete
**Confidence:** HIGH for privacy and PoW pitfalls; HIGH for the 01.5-03 Hyperdrive smoke result; MEDIUM for Phase 2 ingest query behavior until deployed.

## Critical Pitfalls

### Pitfall 1: Quasi-Identifier Fingerprinting

**Risk:** Model, tier, harness, region, token windows, and bucket time can identify a user in combination even when no account or IP is stored.

**Avoidance:**

- Suppress public R2 cells with `n < 5`.
- Log-bin token counts before public emission.
- Keep `tz_offset`, country, `event_id`, and nonce out of public R2 files.
- Never write per-event timing logs.
- Trim outliers per cohort before percentile aggregation.

### Pitfall 2: PoW Contract Drift

**Risk:** Any future reimplementation can misread the byte contract and accept or reject the wrong PoW.

**Avoidance:**

- Keep `spec/pow-v1.md` as the exact 72-byte contract: `challenge_id || payload_hash || nonce`.
- Keep `spec/pow-fixtures.json` generated by `cargo xtask gen-fixtures`.
- Run `cargo test -p pow --locked` and `cargo xtask gen-fixtures --check` on every PR.
- Treat any change to K, challenge format, HMAC, payload hash, nonce encoding, or JCS bytes as a versioned spec change.

### Pitfall 3: JCS Canonicalization Drift

**Risk:** Canonical JSON implementations differ on number formatting, key ordering, escaping, and Unicode preservation.

**Avoidance:**

- Use exactly `serde_jcs = "0.2"` through `crates/event-schema::canonical_bytes`.
- Gate RFC 8785 behavior with `crates/event-schema/tests/jcs_conformance.rs`.
- Keep payload fixture vectors for key ordering, Unicode preservation, number formatting, empty payload, and max-size payload.
- If the KAT gate fails, flip the workspace dependency mechanically to `serde_json_canonicalizer = "0.3"` and rerun fixture drift.

### Pitfall 4: Hyperdrive Pooler Prepared-Statement Behavior

**Risk:** Query APIs that prepare named statements can fail through Hyperdrive's pooler across repeated requests.

**Resolution Status (post-01.5-03 staging deploy):**

Resolved for the current smoke path. Upstream `tokio-postgres` rev `35a85bdbfeeac465e092950f65a10d9192418175` works through Hyperdrive when the Worker uses `query_typed_one` for the constant `SELECT 1` smoke query. Plan 01.5-03 first saw a repeated-request failure with the normal query helper, then switched to `query_typed_one`, redeployed, and repeated `/db-ping` returned HTTP 200 with `{"ok":true}`.

**Avoidance for Phase 2:**

- Prefer `query_typed_*` APIs for Hyperdrive-backed Worker queries.
- Keep Postgres clients per request; no global client or pool in module scope.
- Drive the connection future with `wasm_bindgen_futures::spawn_local`.
- Drop the client before the response future resolves.
- Re-evaluate dependency source only if Phase 2 needs an API shape that cannot avoid prepared statements.

### Pitfall 5: Worker I/O Lifecycle Leaks

**Risk:** Worker invocations can leak sockets or cross request I/O if clients are stored globally.

**Avoidance:**

- Open the Postgres client inside the request handler.
- Spawn the connection future for that request only.
- Do not store clients in `static`, `thread_local!`, `OnceCell`, or lazy globals.
- Keep transactions short and local to the DB operation.
- Return errors without logging credentials or per-event identifiers.

### Pitfall 6: Logging Boundary Leaks

**Risk:** Debug logging can reveal event IDs, nonces, IPs, solve timing, Worker secrets, or Hyperdrive credentials.

**Avoidance:**

- Use the `.github/workflows/pow.yml` `log-boundary` grep gate.
- Do not log `event_id`, `nonce`, `cf-connecting-ip`, elapsed solve/submit timing, password fields, connection strings, or Hyperdrive typed binding fields.
- Treat public Worker logs as semi-public operational data.

### Pitfall 7: Rate Limit Privacy Confusion

**Risk:** Edge throttling by IP can be mistaken for app-level IP persistence.

**Avoidance:**

- Let Cloudflare `RateLimiter` hold counters at the edge.
- Derive keys from `cf-connecting-ip` only for the limiter call.
- Do not log, persist, or include IPs in DB rows or R2 files.
- Return a generic 429 body with route and retry seconds only.

### Pitfall 8: Public R2 Write-Order Races

**Risk:** Dashboard sees a new manifest before bucket files or status files are available.

**Avoidance:**

- Write bucket files first.
- Write `_status.json`.
- Write `manifest.json` last.
- Set immutable cache headers only on rolled-over bucket files.
- Keep manifest TTL short enough for the 15-minute cadence.

### Pitfall 9: Outlier Trim on Skewed Token Data

**Risk:** Sigma-based trimming destroys real right-tail usage or fails to remove attack payloads.

**Avoidance:**

- Use per-cohort double-MAD at `3 * MAD`.
- Compute upper and lower MAD separately.
- Surface trim rate.
- Investigate trim rate over 5%; escalate over 10%.

### Pitfall 10: Open-Source CLI Abuse Defense Theater

**Risk:** Trying to make the canonical CLI trusted through hidden secrets or obfuscation contradicts open source and fails technically.

**Avoidance:**

- Assume clients can submit bogus payloads.
- Enforce closed enum values and bounded token counts server-side.
- Use PoW for economic friction.
- Use edge rate limiting without app IP persistence.
- Use k-anonymity and robust trimming at materialization.
- Document the "we do not trust your CLI either" model in launch docs.

## Phase 2 Checklist

- Use `crates/event-schema::EventPayload` for payload validation.
- Use `crates/pow::verify` for PoW.
- Use workers-rs 0.8.1 `Hyperdrive` typed binding and `query_typed_*` APIs.
- Use workers-rs 0.8.1 `RateLimiter` typed binding.
- Keep DB clients per request.
- Run log-boundary grep after any Worker logging change.
- Keep `/db-ping` public exposure under review before launch.
